# Glossary

Quick reference for terms used throughout the Access Broker documentation.

## Core Concepts

### Claims

Custom data stored in a user's JWT token (specifically in `app_metadata`). Claims are embedded in the token and can be read without a database query.

```typescript
// Reading claims from a user object
const role = user.app_metadata?.apps?.['my-app']?.role; // 'admin', 'user', etc.
```

**Why use claims?** Fast authorization checks without database queries. RLS policies can read claims directly from the JWT.

### app_metadata vs user_metadata

| Field | Who can modify | Use for |
| --- | --- | --- |
| `app_metadata` | Server only (service role key) | Roles, permissions, claims — things users shouldn't change |
| `user_metadata` | User can modify | Profile info like name, avatar, preferences |

### APP_ID

A unique string identifier for an application in a multi-app environment. Used to namespace claims.

```typescript
const APP_ID = 'my-app';
// Claims are stored at: app_metadata.apps['my-app']
```

### Service Role Key

A secret Supabase key that bypasses Row Level Security (RLS). **Never expose to the client.** Only use in server-side code.

```typescript
// Server-side only!
const supabase = createClient(url, process.env.SUPABASE_SERVICE_ROLE_KEY);
```

### Anon Key (Public Key)

A Supabase key that respects RLS policies. Safe to use in client-side code.

```typescript
// Safe for client-side
const supabase = createClient(url, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY);
```

## Authentication Terms

### JWT (JSON Web Token)

A signed token containing user data. Supabase Auth issues JWTs after login. Claims are embedded in the JWT's payload.

### RLS (Row Level Security)

PostgreSQL security policies that control data access at the database level. Can reference JWT claims for authorization.

```sql
-- Example: Only allow users to see their own data
CREATE POLICY "Users see own data" ON documents
  FOR SELECT USING (user_id = auth.uid());
```

### OAuth

Authentication using third-party providers (Google, GitHub, etc.). User logs in via the provider, Supabase creates/links the account.

### Passkeys (WebAuthn)

Passwordless authentication using biometrics (Face ID, Touch ID, fingerprint) or security keys. More secure than passwords.

### MFA (Multi-Factor Authentication)

Requiring a second verification method (like a TOTP code from an authenticator app) in addition to the password.

### TOTP (Time-based One-Time Password)

6-digit codes generated by authenticator apps (Google Authenticator, 1Password, etc.) that change every 30 seconds.

### Magic Link

A passwordless login method where users click a link sent to their email instead of entering a password.

## Architecture Terms

### Access Broker

This project. A platform that provides centralized authentication, authorization, and user management for Supabase applications.

### Auth Portal

The login/signup pages that users interact with. Can be used as a central SSO hub for multiple applications.

### Admin Dashboard

The web interface for administrators to manage users, apps, roles, and permissions.

### SSO (Single Sign-On)

Allowing users to log in once and access multiple applications without re-authenticating.

## Authorization Terms

### RBAC (Role-Based Access Control)

Authorization based on user roles (admin, editor, viewer, etc.). Each role has a set of permissions.

```typescript
if (user.app_metadata?.apps?.['my-app']?.role === 'admin') {
  // Allow admin action
}
```

### ABAC (Attribute-Based Access Control)

Authorization based on user attributes (department, location, subscription tier, etc.). More flexible than RBAC.

### Permissions

Specific actions a user can perform (read, write, delete, invite_users, etc.). Often combined with roles.

```typescript
const permissions = user.app_metadata?.apps?.['my-app']?.permissions;
if (permissions?.includes('delete')) {
  // Allow delete action
}
```

## Supabase-Specific Terms

### Supabase Client

The JavaScript library for interacting with Supabase services (`@supabase/supabase-js`).

### Supabase SSR

The library for server-side rendering with Supabase (`@supabase/ssr`). Handles cookies and sessions in Next.js.

### RPC (Remote Procedure Call)

Calling PostgreSQL functions from client code.

```typescript
await supabase.rpc('set_app_claim', {
  uid: userId,
  app_id: 'my-app',
  claim: 'role',
  value: 'admin',
});
```

### auth.users

The Supabase table containing user accounts. `raw_app_meta_data` column stores claims.
